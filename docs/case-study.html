Refactoring the SDLC out of a heavy plugin

The best-practices theory applied to one real system: a token-heavy Claude Code plugin rebuilt as two lean, plugin-free local skills — every bit of process discipline kept, per-message cost cut by roughly two-thirds.

Read the theory doc first if "tier 1/2/3" and "idle vs. per-message cost" aren't already familiar — this doc uses that vocabulary throughout.

The starting point

The lifecycle lived in a plugin, dev-command-center, in a single skill, dev-orchestrator. It worked — but it was expensive in exactly the ways the theory predicts:

  • A monolithic skill body covering the entire lifecycle plus a board/tracking subsystem — measured at ~8,392 tokens. Because it covered everything, nearly every dev session triggered it, so that weight sat in sticky tier 2 for whole sessions.
  • Four registered agents — four tier-1 description lines, paid every turn of every session, review or no review.
  • A board + GitHub-Pages tracker, public-board privacy rules, an onboarding scanner, a self-improvement auditor loop, and an autonomous-mode section — all carried in the body.

The reviews were the runtime killer: spelled out as full multi-agent panels, a single slice could spin up 25–38 review agents. Correct, but wildly out of proportion to most slices.

What we were actually optimizing

We did not set out to delete capability — the user liked the lifecycle. The goal was to keep the discipline and move the weight off the hot path, via three moves: cut unused machinery; re-implement reviews so the default path is cheap and escalates only when warranted; and split the monolith along the seam the user already works on.

The moves

Move 1 — Delete the board subsystem

The user had stopped tracking on the board, so all board machinery came out. 8,392 → 4,929 tokens (~41% off the sticky body), zero loss of discipline — the board was never part of the discipline. Its one useful function, resume-after-close, was replaced by a far cheaper artifact: a per-project docs/dev-state.md a fresh session reads in ~1 KB instead of reloading a transcript. (Best practice #5.)

Move 2 — Fold the reviews down to their cheap default

All three reviews stayed; their form now scales to stakes (a "rigor dial"): plan-check and merge-gate became single inline subagents, with the merge-gate escalating to the full multi-lens panel only for high-stakes slices; health-sweep stayed an advisory workflow. In body tokens this barely moved (4,929 → 4,882) — but a normal slice went from ~25–38 review agents to about 2. The heavy reading still happens, just in tier-3 isolated contexts that return only a verdict. (Best practices #3, #4.)

Move 3 — Split the monolith into orchestrator + builder

The lifecycle has a natural seam: a long-lived conductor and an ephemeral executor. We split along it so each session loads only its half.

The builder is genuinely ephemeral — its heavy implementation log dies on teardown and never reaches the conductor. (Best practice #6.)

Move 4 — Inline subagents instead of registered agents

The four registered agents are gone; reviewer rubrics now live inside the orchestrator's body and dispatch as inline subagents. Idle cost dropped from four always-on agent descriptions to ~196 tokens for the two skill descriptions — same isolation, same review independence, zero idle tax. (Best practice #4.)

Move 5 — Cut the dead weight

Removed because unused and body-only: the self-improvement auditor loop, the autonomous/supervised-remote section (Claude Code does this natively), the onboarding scanner, and the public-board privacy rules. Kept deliberately: "compose, don't reinvent" and the rigor dial — both small and load-bearing.

The lifecycle is identical from the user's seat: same stages, same hard gates, same three reviews, same stack gates. What changed is purely where the weight sits.

What we deliberately did not do

  • Didn't move agents to a remote runner to "save context." It wouldn't — a local subagent already keeps its context out of the parent. Hosting changes where compute runs, not what's in the prompt.
  • Didn't keep it as a plugin "to be safe." A plugin isn't the problem — registration weight is. A lean two-skill, zero-agent plugin would cost nearly the same at idle. We went plugin-free for clean install/versioning, not for token reasons.
  • Didn't over-optimize cold artifacts. Workflow scripts and human docs cost nothing until used, so we left them readable.

The transferable lesson

Keep the always-loaded and sticky paths small; push heavy reading into isolated contexts; resume from cheap artifacts. The lifecycle didn't get weaker — it got cheaper, because the cost was never in the discipline, it was in where the discipline was stored.
