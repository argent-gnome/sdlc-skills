The house SDLC

One build process, run the same way every time, by three cooperating Claude Code skills — a plugin-free, token-lean successor to the old dev-command-center plugin.

Why three skills

Every message re-sends the whole session context, so a loaded skill body is a tax paid per message. A single monolithic SDLC skill makes every session pay for the whole lifecycle. Splitting along the seam you already work on means each session loads only its half.

shaper

house-shaper

A user-run shaping session for the fuzzy front end — research, brainstorm, spec, plan, plan-check, doc reconcile. Turns an idea into ready-to-build work (or a recorded decision), then hands off. Torn down after.

conductor

house-orchestrator

One long-lived, resumable session. Sequences the slice, holds the gates, writes kickoff prompts, dispatches reviews to subagents, reconciles.

executor

house-builder

Ephemeral. Implements ONE plan unit — TDD, stack gates, self-review, doc reconcile — reports back, then is torn down (its heavy log dies with it).

The orchestrator stays light because it never reads diffs itself: reviews run in subagents, and only the verdict (a few hundred tokens) returns.

⛔ hard gate — the orchestrator stops and gets the user: spec review, mockup sign-off, live validation, CI red, any plan deviation, any irreversible/outward-facing action. These never auto-clear, even when running unattended.

The three reviews

reviewwhenwhat
plan-check (4¼)before codeone subagent critiques the plan vs app+spec through 5 lenses → must-fix + advisory. Cheapest place to catch a design flaw.
merge-gate (7)before mergeindependent refute-biased subagent on the diff → GO/NO-GO. The rigor dial escalates to the multi-lens panel for high-stakes slices.
health-sweep (7½)after GOadvisory whole-app workflow → ranked debt backlog. Never blocks; need not run every slice.

Reviews are subagents/workflows, never inline — the heavy reading happens in throwaway context, and independence (eyes that never saw the build) is the whole point of the merge-gate.

Resume

The orchestrator is closeable and reopenable. It rebuilds its picture from a per-project docs/dev-state.md — a short plaintext tracker (active slice, stage, next action, blocked-on, slated, done) updated at transitions and session end. You resume by starting a fresh session that reads this file, not by reloading a heavy transcript.

Docs & hygiene doctrine

The skills share one runtime reference — references/doctrine.md in the orchestrator — read on-demand (never preloaded) at resume, finish, reconcile, and any doc write. It defines four things:

  • Doc model — one job per doc: dev-state.md (operational tracker), roadmap.md (durable spine), docs/adr/ (decision records — why), specs/plans (per-slice design authority), retros, docs/health/.
  • dev-state allowlist — dev-state.md is operational only; durable strategy and decisions are banned and route to roadmap/ADR. The test: if it's still true three slices from now, it isn't dev-state.
  • Routing — a decision → an ADR; a scope change → roadmap; progress → dev-state; as-built drift → reconcile the spec/plan in the slice PR.
  • Hygiene — per-merge the branch/worktree/stash are torn down; resume runs a git-reality check (squash-merge caveat: git branch --merged misses squash-merged branches — confirm via PR state).

All of these are wired into the loop — the resume git-reality check and repo-setup readiness at stage 2, per-merge teardown at finish, and the dev-state lint + broad hygiene sweep at reconcile — under a conservative auto-fix boundary: only provably-safe cleanups happen automatically; anything destructive is surfaced for the user, never silent.

Dropped vs the old plugin

The board + GitHub-Pages tracker, public-board privacy rules, onboarding scanner, self-improvement auditor loop, autonomous-mode setup, and the plugin/marketplace machinery — all removed or replaced by the lightweight dev-state.md. The lifecycle discipline stayed; the token weight didn't.
