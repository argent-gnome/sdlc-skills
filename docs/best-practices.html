Token-lean Claude Code: skills, subagents, and context

A practitioner's guide to why a Claude Code setup gets expensive and how to make it cheap without losing capability. The companion case study shows it applied to the house SDLC skills.

Context is paid for per message, not per session. Anything loaded into a session is re-sent on every turn for the rest of that session. The whole craft is moving weight out of the always-loaded path and into places paid for only when actually used.

Why this matters

A language model is stateless — it doesn't remember your conversation between turns. To continue, the entire context (system prompt, tool defs, every prior message, every loaded skill body) is re-sent on every turn. So:

A 3,000-token skill body loaded for a 60-message session isn't a 3,000-token cost. It's re-sent ~60 times. The thing you load once is paid for continuously.

The expensive question is never "how big is this file." It's "how long does it stay in context, and how many turns pay for it." A big file used once and discarded is cheap. A small file that's always present is, over a long session, surprisingly expensive.

How context works in an agent session

Picture a session's context as three tiers, ordered by what they cost you.

tier 1 — always loaded, every turn

The expensive path

System prompt, every tool definition, and every registered skill/agent description (the one-line metadata, not the body), plus the growing transcript. This is "idle cost" — paid whether or not you use it. A registered agent you never invoke still costs its description line on all 100 turns.

tier 2 — loaded on demand, then sticky

The per-message path

Invoking a skill loads its full body — and it stays loaded to the session's end, re-sent each turn. This tier quietly dominates cost. A monolithic 8,000-token skill invoked early in a long session is the single most expensive thing in most setups.

tier 3 — isolated, returns only a result

The nearly-free path

A subagent runs in its own context. It reads files and reasons at length — none of it lands in your session. Only its final output (a few hundred tokens) returns. The 50,000 tokens of diffs it read to produce a GO/NO-GO die with its context. This is the lever.

The plugin trap

Plugins aren't inherently heavy — but how they're usually built makes them heavy. A plugin ships two kinds of things:

  • Skills — each adds a description to tier 1 (idle), and its body to tier 2 only if invoked.
  • Registered agents — each adds a description to tier 1 always, because the model must know the agent exists to delegate to it.

So a plugin with one lean skill and zero agents is nearly free at idle. A plugin with a monolithic skill and four agents pays four agent description lines every turn forever, plus the full skill body every turn once anything triggers it — and a whole-lifecycle skill triggers in nearly every session.

Registration is the cost. Anything that must announce itself in tier 1 is an idle tax. The number of registered surfaces — agents especially — is what you pay at idle; a fat skill body is what you pay per-turn once it's live.

Subagents vs. loaded ("full") agents

A registered agent and an ad-hoc subagent are the same execution mechanism. Both run in an isolated context and return only output; neither keeps its working context in your session. The only difference is registration:

registered agentone always-on description linein the agent definition
inline subagent (dispatched with a prompt)zeroin the body of the skill that dispatches it

Keep a reviewer's rubric inside the dispatching skill's body and spawn an inline subagent with it, and you pay zero idle cost — the rubric is only in context when its owning skill is already loaded.

Moving an agent to a remote/hosted runner does not reduce context load. A local subagent already keeps its context out of the parent. Hosting changes where the CPU runs, not what's in your prompt.

The best practices

1. Minimize the always-loaded surface, not total capability. Count your tier-1 lines (tool defs, skill + agent descriptions). That number is multiplied by every turn of every session. Capability in tier 2/3 is comparatively free.
2. Split a monolith along the seam you actually work on. A whole-lifecycle skill taxes every session for parts it isn't using. One skill per role (conduct vs. execute, plan vs. build) means each session loads only its half.
3. Put heavy reading in subagents; keep only verdicts in the parent. Reviews, audits, diff analysis, log trawls — anything that reads a lot to conclude a little. The parent receives the conclusion, never the raw material.
4. Prefer inline subagents over registered agents for rubric work. If a reviewer's instructions can live in the dispatching skill's body, you avoid the idle tax entirely. Register an agent only when it must be invokable independently of any skill.
5. Make long-lived sessions resumable from a cheap artifact. A conductor you keep reopening shouldn't reload a giant history. Have it write a short plaintext state file and rebuild from that — a 1-KB status file beats dragging a 100-message transcript forward.
6. Make ephemeral sessions actually ephemeral. A build session's heavy log is its biggest cost. Spin it up for one unit, tear it down, and the log dies with it instead of contaminating the long-lived conductor.
7. Don't optimize what isn't on the hot path. A doc read once is cheap; a workflow script on disk costs nothing until it runs. Spend effort on the always-loaded tier and the sticky tier-2 bodies — that's where the multiplier lives.

A quick mental model

Before adding a capability, ask in order:

  1. Does it need to announce itself at idle? If not, don't register it — dispatch inline.
  2. Does it read a lot to conclude a little? If so, it's a subagent, not inline skill text.
  3. Will it stay loaded once triggered? If so, is it carrying weight only some sessions need? Split it.
  4. Is this a long-lived session? Then give it a cheap resume artifact so reopening is free.

Get those four right and a setup that felt heavy becomes lean without losing a single capability — which is exactly what the case study walks through.
